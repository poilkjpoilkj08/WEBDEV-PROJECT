1. OVERVIEW
================================================================================

The BookHive shipping system uses a distance-based formula to calculate 
realistic shipping costs for multiple Indonesian couriers.

FORMULA (Applied):
─────────────────────────────────────────────────────────────────
SHIPPING COST = Base Fee + (Distance_km × Price_per_km) + (Weight_kg × Price_per_kg)

Where:
  • Base Fee: Fixed starting cost per courier (IDR)
  • Distance_km: Real distance from store to customer (Haversine formula)
  • Price_per_km: Distance-based charge per courier
  • Weight_kg: Total weight of all books in order (converted from grams)
  • Price_per_kg: Weight-based charge per courier

Distance Calculation Method:
  Haversine Formula calculates the great-circle distance between two points
  using their latitude and longitude coordinates. This provides accurate,
  real-world shipping distances for Indonesia's geography.

4. IMPLEMENTATION IN CODEBASE
================================================================================

Backend (app/Services/ShippingService.php):
─────────────────────────────────────────────────────────────────

The service implements the formula as:

    public function calculateShippingCost($distance_km, $weight_grams, $method)
    {
        $rates = self::COURIER_RATES[$method];
        $weight_kg = $weight_grams / 1000;
        
        // Apply formula
        $cost = $rates['base_fee'] 
              + ($distance_km * $rates['price_per_km'])
              + ($weight_kg * $rates['price_per_kg']);
              
        return (int)round($cost);
    }


Frontend (resources/views/checkout/checkout.blade.php):
─────────────────────────────────────────────────────────────────

1. Collect weight from cart items:
   const totalWeightGrams = document.querySelectorAll('.cart-item')
       .reduce((sum, item) => {
           const weight = parseInt(item.dataset.weightGrams);
           const qty = parseInt(item.dataset.quantity);
           return sum + (weight * qty);
       }, 0);

2. Send calculation request with weight:
   axios.post('/checkout/calculate-shipping', {
       store_id: storeId,
       latitude: customerLat,
       longitude: customerLng,
       weight_grams: totalWeightGrams,
       method: courierCode
   });

3. Display cost breakdown:
   Cost: 287,650 IDR
   Breakdown: 180km @ 1,500/km + 1.13kg @ 5,000/kg = 277,650
   Per-km: 1,543 IDR/km


Controller (app/Http/Controllers/CheckoutController.php):
─────────────────────────────────────────────────────────────────

The calculateShipping() endpoint:
  1. Validates input (store_id, coordinates, weight, method)
  2. Gets store coordinates from database
  3. Calculates distance using Haversine formula
  4. Calls ShippingService->calculateShippingCost()
  5. Returns JSON with cost, breakdown, per-km rate

Response Format:
    {
        "success": true,
        "cost": 287650,
        "distance": 180.5,
        "per_km_rate": 1542,
        "breakdown": {
            "distance_km": 180.5,
            "weight_kg": 1.13
        }
    }

Problem 1: Shipping cost not showing during checkout
─────────────────────────────────────────────────────────────────
Check:
  1. Browser Console (F12): Any JavaScript errors?
  2. Network Tab: API request to /checkout/calculate-shipping?
  3. Server Logs: tail -f storage/logs/laravel.log
  4. Database: Is store_locations table populated with coordinates?

Solution:
  • Verify store has latitude/longitude values
  • Check that cart items have weight_grams in books table
  • Verify checkout.blade.php has .cart-item elements

Problem 2: Shipping costs seem incorrect
─────────────────────────────────────────────────────────────────
Debug Steps:
  1. Note the cost shown: e.g., 287,650
  2. Note the courier: e.g., "J&T Regular"
  3. Check API response (Network tab or logs)
  4. Manual calculation: base + (distance×rate_per_km) + (weight×rate_per_kg)

Common Issues:
  • Distance not calculated correctly (check Haversine formula)
  • Weight not converted to kg (should divide by 1000)
  • Wrong courier rates in ShippingService
  • Book weight_grams is NULL (check database)

Problem 3: Same route shows different prices each load
─────────────────────────────────────────────────────────────────
Status: NORMAL
Reason: Different stores or slight coordinate variations
Solution: This is expected. Always calculate fresh based on user input.

Problem 4: Calculations differ between online and spreadsheet
─────────────────────────────────────────────────────────────────
Verify:
  1. Are using same base fees and per-km/per-kg rates?
  2. Is distance calculated the same way (Haversine)?
  3. Is weight calculated as sum of all books × qty?

Use Tinker to test:
    php artisan tinker
    > $service = new App\Services\ShippingService;
    > $cost = $service->calculateShippingCost(180, 1130, 'jne_reg');

5. FALLBACK CALCULATION (When API is Unavailable)
================================================================================

If Binderbyte API fails or is unreachable, the system uses local calculation:

Formula:
─────────────────────────────────────────────────────────────────
Final Cost = Base Cost + (Weight × Weight Cost)

Where:
  - Base Cost: Courier's flat base rate (see section 2)
  - Weight: Book weight in kilograms
  - Weight Cost: IDR 2,000 per kilogram

Example Calculation:
─────────────────────────────────────────────────────────────────
Courier: JNE Regular (jne_reg)
Base Cost: 15,000 IDR
Weight: 1 kg
Weight Cost per kg: 2,000 IDR

Final Cost = 15,000 + (1 × 2,000) = 17,000 IDR

Calculation for 2kg book:
Final Cost = 15,000 + (2 × 2,000) = 19,000 IDR

================================================================================

API calls are cached for efficiency:

Example Cache Key:
─────────────────────────────────────────────────────────────────

This means if a user calculates the same route with the same courier within
24 hours, the cached result is returned instead of making a new API call.

Benefits:
  ✓ Reduces API call costs
  ✓ Faster response times
  ✓ Reduces load on Binderbyte servers
  ✓ Better user experience during checkout

Cache is automatically invalidated after 24 hours, fetching fresh rates.

7. CALCULATION IN CHECKOUT FLOW
================================================================================

During checkout, shipping cost calculation happens in this order:

Step 1: User selects shipping method
        ↓
Step 2: System calls /api/checkout/calculate-shipping endpoint
        ↓
Step 3: CheckoutController::calculateShipping() is invoked
        ↓
Step 4: ShippingService attempts Binderbyte API call
        ├─ API Success? → Return API cost (Binderbyte calculated)
        └─ API Failed? → Use fallback calculation (local formula)
        ↓
Step 5: Cost displayed to user in checkout form
        ↓
Step 6: User confirms order with selected method
        ↓
Step 7: Cost stored in orders table as 'shipping_cost'

Location Information Used:
  - Store Location: From store_locations table
  - Customer Location: From checkout form (city and postal code)
  - Weight: Default 1kg (can be enhanced to actual book weight)

13. TROUBLESHOOTING
================================================================================

Problem: Shipping cost not appearing during checkout
─────────────────────────────────────────────────────────────────
Solutions:
  1. Check browser console for JavaScript errors
  2. Verify /api/checkout/calculate-shipping endpoint responds
  3. Check logs: tail -f storage/logs/laravel.log
  4. Verify ShippingService is properly imported in CheckoutController
  5. Ensure Binderbyte API key is correct

Problem: Shipping costs are suspiciously low/high
─────────────────────────────────────────────────────────────────
Likely Cause: Fallback calculation is being used (API failed)
Solutions:
  1. Check laravel.log for API errors
  2. Verify API key is valid and active
  3. Check Binderbyte API status
  4. Verify origin and destination cities exist in Binderbyte database
  5. Update fallback weight cost if desired: change 2000 in ShippingService.php

Problem: Same route shows different prices at different times
─────────────────────────────────────────────────────────────────
Expected Behavior: Binderbyte rates update daily
Solution: This is normal. Courier rates fluctuate based on demand
         and fuel costs. The system always fetches latest rates.

Problem: API seems to be timing out
─────────────────────────────────────────────────────────────────
Solutions:
  1. Verify internet connection
  2. Check if Binderbyte API is operational: https://binderbyte.com
  3. Verify firewall allows HTTPS requests to api.binderbyte.com
  4. Check network tab in browser dev tools for timeout details
  5. Increase HTTP timeout in CheckoutController if needed
